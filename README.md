## v1.0.0 -- The manifesto is ready

AI systems are beginning to remember us -- our preferences, goals, habits, vulnerabilities, and the patterns they infer from everything we say and do. This memory is becoming a new layer of digital identity, and right now it is being built almost entirely on the provider's terms.

**The AI Memory Manifesto** argues that AI memory is a distinct category of data that deserves its own rights, standards, and protections. After months of drafting, sourcing, cross-referencing, and community review, we are publishing v1.0.0.

### What the manifesto contains

- **16 enumerated rights** governing AI memory -- from the right to know and access, through portability and deletion, to freedom from memory-based manipulation
- **12 principles** for an open memory standard -- user sovereignty, interoperability, transparency, distinguishability, reversibility, and more
- **A technical agenda** -- open formats, portable schemas, user-owned vaults, cryptographic integrity, permission layers for agents
- **A political claim** -- AI memory is not telemetry, not product state, not an optimization artifact. It is a new category of user data that demands its own governance

### Why it matters now

Every major AI provider is building persistent memory. The infrastructure is being decided right now -- memory formats, retention defaults, portability (or lack of it). The norms are not yet fixed. The dominant architectures are not yet inevitable.

If we wait until memory lock-in is irreversible, user control will be much harder to win.

This is the window.

### What changed since draft stage

- **41 primary sources** verified and linked across all 8 provider sections in the evidence document
- **Citation standard** established -- every factual claim requires an official announcement, policy page, regulatory decision, or peer-reviewed publication
- **Anthropic section corrected** -- training policy updated to opt-in model (2025), tier classifications fixed, retention periods corrected
- **Terminology formalized** -- 30+ definitions with academic references and legal grounding (GDPR, NIST, EU AI Act)
- **Landscape mapped** -- 6 open standards/protocols, 5 open-source projects, regulatory frameworks across EU and US, 8 policy/research organizations
- **Distinguishability reconciled** to 5 canonical memory categories applied consistently across all documents
- **Interactive website launched** at [antonandrusenko.github.io/ai-memory-manifesto](https://antonandrusenko.github.io/ai-memory-manifesto/) with the full manifesto, dark/light themes, and a visual demonstration of memory lock-in

## Who this is for

- **Builders** -- if you're designing memory systems, agent architectures, or personal data stores, the 16 rights are a design checklist
- **Researchers** -- if you work on inference rights, cognitive sovereignty, or AI governance, the manifesto names the gap between existing frameworks and AI memory
- **Designers** -- if you shape how people interact with AI, memory visibility and control are unsolved UX problems
- **Lawyers and policymakers** -- if you work on data protection, the manifesto maps where GDPR, the EU AI Act, and US state laws fall short on AI memory
- **Product managers** -- if you own a memory feature, the evidence document shows what your competitors do and where the industry is headed
- **Users** -- if you use ChatGPT, Claude, Gemini, Copilot, or any AI with persistent memory, this is about your data and your rights

## Open questions

These are unresolved problems we want the community to help think through:

1. **Derivative memory governance** -- No provider explicitly addresses deletion or retention of summaries, embeddings, and inferences derived from user data. How should derivative memory be governed?
2. **Inference portability** -- GDPR Article 20 explicitly excludes inferred data from portability. Should a memory portability standard include inferences?
3. **Memory across agents** -- As AI agents act on our behalf across services, who governs the memory they accumulate? The user? The agent developer? The service provider?
4. **Consent granularity** -- Is per-memory-item consent practical, or do we need a better model for memory permissions?
5. **Competing standards** -- PAM, AMP, and proprietary approaches (Google, Anthropic) are diverging. How do we converge toward interoperability before lock-in hardens?
6. **Cognitive sovereignty** -- How do we operationalize the academic concept of cognitive sovereignty into enforceable technical and legal requirements?

If any of these interest you, [open an issue](https://github.com/antonandrusenko/ai-memory-manifesto/issues) or submit a pull request.

## Related initiatives

This manifesto is aligned with a growing ecosystem working on AI memory governance:

- [Open Memory Alliance](https://openmemoryalliance.org/) -- open APIs and user-first policies
- [Portable AI Memory (PAM)](https://github.com/portable-ai-memory/portable-ai-memory) -- vendor-neutral JSON interchange format
- [AI Memory Protocol (AMP)](https://github.com/purmemo-ai/purmemo-amp) -- normalizing conversation history exports
- [Mem0 / OpenMemory](https://github.com/mem0ai/mem0) -- open-source local AI memory infrastructure
- [New America OTI](https://www.newamerica.org/oti/briefs/ai-agents-and-memory/) -- policy analysis on AI agents and memory
- [IETF AIPREF Working Group](https://datatracker.ietf.org/wg/aipref/about/) -- standardizing AI data processing preferences
- [Data Transfer Initiative](https://dtinit.org/) -- conversation history portability

For the full map, see [Landscape](docs/landscape.md).
